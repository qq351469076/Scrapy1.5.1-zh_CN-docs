Scrapy shell

Scrapy shell是一个交互式shell，您可以在其中快速尝试和调试抓取代码，而无需运行爬行器。它本来是用来测试数据提取代码的，但实际上您可以用它来测试任何类型的代码，因为它也是一个常规的Python shell。

shell用于测试XPath或CSS表达式，并查看它们是如何工作的，以及它们从您试图获取的web页面中提取了哪些数据。它允许您在编写爬行器时交互式地测试表达式，而无需运行爬行器来测试每个更改。

一旦熟悉了Scrapy shell，您就会看到它是开发和调试爬行器的非常有用的工具。

配置shell

如果已经安装了IPython，那么Scrapy shell将使用它(而不是标准的Python控制台)。IPython控制台功能更加强大，提供了智能自动完成和彩色输出等功能。

我们强烈建议您安装IPython，特别是如果您正在使用Unix系统(IPython擅长的领域)时。有关更多信息，请参阅IPython安装指南。

Scrapy还支持bpython，并将在IPython不可用的地方尝试使用它。

通过scrapy的设置，您可以将其配置为任何一个ipython、bpython或标准pythonshell，而不管安装了哪个。这是通过设置SCRAPY_PYTHON_SHELL环境变量来实现的;或者在scrapy.cfg中定义它。

[settings]
shell = bpython

启动shell

要启动Scrapy shell，可以使用shell命令如下:

scrapy shell <url>

其中url是您想要抓取的页面。

shell也适用于本地文件。如果您想使用一个web页面的本地副本，这可能非常方便。shell对本地文件理解如下语法:

# UNIX-样式
scrapy shell ./path/to/file.html
scrapy shell ../other/path/to/file.html
scrapy shell /absolute/path/to/file.html

# 文件URI
scrapy shell file:///absolute/path/to/file.html

在使用相对文件路径时，要显式地在它们前面加上`./`(或`../`)。
`scrapy shell index.html`不会像预期的那样工作(这是设计出来的，不是bug)。
因为`shell`更喜欢HTTP url而不是文件uri和`index.html`, 在语法上与`example.com`类似
`shell`将处理`index.html`作为一个域名，并触发一个DNS查找错误:

$ scrapy shell index.html
[ ... scrapy shell starts ... ]
[ ... traceback ... ]
twisted.internet.error.DNSLookupError: DNS lookup failed:
address 'index.html' not found: [Errno -5] No address associated with hostname.

如果文件名为`index.html`并且存在于当前目录中, `shell`将不会预先测试。再次,要明确。

使用shell

Scrapy shell只是一个常规的Python控制台(如果您有它的话，也可以称为IPython控制台)，它提供了一些方便的附加快捷函数。

可用的方法

  • shelp(): 在控制台打印帮助信息以及一系列可用对象的列表和方法
  • fetch(url[, redirect=True]): 从给定的URL获取新的response并相应地更新所有相关对象。您可以选择是否需要HTTP 3xx重定向，以避免后续的传递redirect=False
  • fetch(request): 从给定的请求中获取一个新的response，并相应地更新所有相关对象。
  • view(response): 在本地web浏览器中打开给定的response，以供检查。这将向响应体添加<base> tag，以便外部链接(如图像和样式表)正确显示。然而，请注意，这将在您的计算机中创建一个临时文件，它不会被自动删除。

可用的Scrapy对象

Scrapy shell自动从下载的页面创建一些方便的对象，比如Response对象和Selector对象(HTML和XML内容)。

这些对象是:

  • crawler: 当前Crawler对象。
  • spider: 已知可以处理URL的Spider，或者如果没有为当前URL找到爬行器，则为当前为Spider对象
  • request: 一个Request对象用于获取最新的页面。您可以使用replace()修改此请求，或者使用fetch快捷方式获取新请求(不离开shell)。。
  • response: 一个Response对象包括最后获取的页面
  • settings: 当前Scrapy的设置

shell会话示例

下面是一个典型的shell会话示例，我们从抓取https://scrapy.org页面开始，然后继续抓取https://reddit.com页面。最后，我们修改了(Reddit)请求方法来用POST重新取回它，并得到一个错误。我们通过键入Ctrl-D(在Unix系统中)或Ctrl-Z来结束会话。

请记住，这里提取的数据在您尝试时可能不是相同的，因为这些页面不是静态的，并且在您进行测试时可能已经发生了更改。这个示例的唯一目的是让您熟悉Scrapy shell的工作原理。

调用spiders中的shell来检查responses

有时候，您希望检查在您的爬行器的某个点上正在处理的responses，如果只是为了检查您期望的response是否到达那里。

这可以通过使用scrap.shell.inspect_response函数。

下面是一个例子，告诉你如何从你的spider中调用它:

import scrapy


class MySpider(scrapy.Spider):
    name = "myspider"
    start_urls = [
        "http://example.com",
        "http://example.org",
        "http://example.net",
    ]

    def parse(self, response):
        # 我们想要检查一个特定的response.
        if ".org" in response.url:
            from scrapy.shell import inspect_response
            inspect_response(response, self)

        # 解析代码的其余部分.

当你运行spider时，你会得到类似的结果:

2014-01-23 17:48:31-0400 [scrapy.core.engine] DEBUG: Crawled (200) <GET http://example.com> (referer: None)
2014-01-23 17:48:31-0400 [scrapy.core.engine] DEBUG: Crawled (200) <GET http://example.org> (referer: None)
[s] Available Scrapy objects:
[s]   crawler    <scrapy.crawler.Crawler object at 0x1e16b50>
...

>>> response.url
'http://example.org'

然后，您可以检查提取代码部分是否工作:

>>> response.xpath('//h1[@class="fn"]')
[]

不,它不是。所以你可以在你的web浏览器中打开response，看看它是不是你期望的response:

>>> view(response)
True

最后，按Ctrl-D(或窗口中的Ctrl-Z)退出shell并恢复爬行:

>>> ^D
2014-01-23 17:50:03-0400 [scrapy.core.engine] DEBUG: Crawled (200) <GET http://example.net> (referer: None)
...

注意，这里不能使用fetch快捷方式，因为Scrapy引擎被shell阻塞。然而，在你离开shell后，spider会继续在它停止的地方爬行，如上所示。
