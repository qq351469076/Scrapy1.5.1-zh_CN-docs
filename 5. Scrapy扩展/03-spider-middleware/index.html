Spider Middleware

spider middleware是一个hooks(钩子)的框架，可以插入到Scrapy的spider处理机制当中，在那里您可以插入定制的功能来处理发送给spider的response，并处理来自于spider的requests和items。

激活一个spider middleware

要激活一个spider middleware组件，将它添加到SPIDER_MIDDLEWARES setting中，它的键是middleware类的路径, 并且它们的值是对应的数字。

这有一个例子:

SPIDER_MIDDLEWARES = {
    'myproject.middlewares.CustomSpiderMiddleware': 543,
}

SPIDER_MIDDLEWARES setting与Scrapy中定义的SPIDER_MIDDLEWARES_BASE setting合并(而不是覆盖),然后按顺序排序,得到最终排序的middlewares列表):第一个middleware接近engine,最后一个接近spider。换句话说，每个middleware的process_spider_input()方法将在增加的middleware顺序（100、200、300、。）中被调用，并且每个中间件的process_spider_output()方法将以递减的顺序被调用。

要决定分配给您的middleware的顺序，请参阅SPIDER_MIDDLEWARES_BASE setting，并根据您想要插入middleware的位置选择一个值。这个顺序很重要，因为每个middleware执行不同的操作，您的middleware可能依赖于应用的一些以前（或后续）middleware。

如果您想要禁用内置的middleware（在SPIDER_MIDDLEWARES_BASE中定义的，在默认情况下启用），您必须在项目SPIDER_MIDDLEWARES setting中定义它，并将None赋值为其值。例如，如果您想要禁用off-site middleware：

SPIDER_MIDDLEWARES = {
    'myproject.middlewares.CustomSpiderMiddleware': 543,
    'scrapy.spidermiddlewares.offsite.OffsiteMiddleware': None,
}

最后，请记住，有些middleware可能需要通过特定的setting来启用。查看每个middleware文档以获得更多信息。

编写自己的spider middleware

每个middleware组件都是一个Python类，它定义了下列方法中的一个或多个：

scrapy.spidermiddlewares.SpiderMiddleware 这是一个类(class)

process_spider_input(response, spider)

这个方法被调用来处理通过spider middleware和进入spider的每一个response。

process_spider_input()应该返回None或抛出异常。

如果它返回None，Scrapy将继续处理这个response，执行所有其他的middlewares，直到最后，response被交给spider进行处理。

如果它抛出了一个异常，那么Scrapy就不会调用任何其他的spider middleware的process_spider_input()，并将调用request errback。errback的输出被链接回另一个方向，以便process_spider_output()处理它，或者如果它引发了异常以便process_spider_exception()来处理。

参数:

  • response (Response 对象) – 正在处理的response
  • spider (Spider 对象) – 这个response对应的spider

process_spider_output(response, result, spider)

这个方法在处理完response之后，从spider返回的结果调用。

process_spider_output()必须返回可迭代的Request、字典或Item对象。

参数:

  • response (Response 对象) – 从spider那里产生了这个输出的response

  • result (一个可迭代的Request, dict或Item对象) – spider返回的结果

  • spider (Spider 对象) – spider的结果正在被处理

process_spider_exception(response, exception, spider)

当spider或process_spider_input()方法（来自spider middleware）抛出异常时，就会调用此方法。

process_spider_exception()应该返回None或一个可迭代的Request、dict或Item对象。

如果它返回None，Scrapy将继续处理这个异常，在接下来的middleware组件中执行任何其他process_spider_exception()，直到没有middleware 组件被保留，并且异常到达engine（在那里它被记录和丢弃）。

如果它返回一个可迭代的process_spider_output() pipeline就会启动，那么就不会调用其他process_spider_exception()。

参数:

  • response (Response 对象) – 当异常被抛出时正在处理的response

  • exception (Exception 对象) – 抛出的异常

  • spider (Spider 对象) – 引起异常的spider

process_start_requests(start_requests, spider)

版本0.15添加此功能

这个方法调用了spider start requests，并且类似于process_spider_output()方法，只是它没有相应的response，并且必须返回requests（而不是items）。

它接收一个可迭代（在start_requests参数中），并且必须返回另一个可迭代的Request对象。

当在您的spider middleware中实现此方法时，
你应该总是返回一个可迭代的（在输入1之后）而不是消耗所有的'start_requests'迭代器，
因为它可以非常大（甚至是无限制的）并导致内存溢出。
Scrapy engine的设计目的是在有能力处理它们的时候拉起start requests，
因此start requests迭代器可以有效地无穷无尽，
因为有其他条件可以阻止spider（如时间限制或item/page计数）。

参数:

start_requests (一个可迭代的Request) – start requests

spider (Spider 对象) – start requests所属的spider

from_crawler(cls, crawler)

如果存在的话，这个classmethod被调用来从Crawler创建一个middleware实例。它必须返回middleware的新实例。Crawler对象提供对所有Scrapy 核心组件的访问，如settings和signals;它是middleware访问它们并将其功能连接到Scrapy的一种方式。

参数:

crawler (Crawler 对象) – 使用这个middleware的crawler

DepthMiddleware

scrapy.spidermiddlewares.depth.DepthMiddleware 这是一个类(class)

DepthMiddleware用于跟踪被抓取站点内的每个Request的深度。它通过setting request.meta[‘depth’] = 0来工作。当没有先前设定的值时（通常只是第一个Request），并将其增加1。

它可以用来限制最大深度的抓取，根据它们的深度来控制Request优先级，诸如此类。

DepthMiddleware可以通过下列settings进行配置（请参阅设置文档以获得更多信息）：

  • DEPTH_LIMIT – 允许在任何站点爬行的最大深度。如果是零，就不会有限制。

  • DEPTH_STATS – 是否收集深度stats。

  • DEPTH_PRIORITY - 是否根据他们的深度对requests进行优先级排序

HttpErrorMiddleware

scrapy.spidermiddlewares.httperror.HttpErrorMiddleware 这是一个类(class)

过滤掉不成功(错误)的HTTP responses，使spider不必处理它们，这(大多数情况下)增加了开销，消耗了更多的资源，并使spider逻辑更加复杂。

根据HTTP标准，成功的responses是那些状态码在200-300范围内的responses。

如果您仍然希望处理超出此范围的response代码，可以使用handle_httpstatus_list spider属性或HTTPERROR_ALLOWED_CODES setting指定爬行器能够处理的response代码。

例如，如果您想让spider处理404 responses，可以这样做:

class MySpider(CrawlSpider):
    handle_httpstatus_list = [404]

Request.meta的handle_httpstatus_list键还可以用于指定每个request允许的response代码，如果您希望对request允许任何response代码，还可以将meta键handle_httpstatus_all设置为True。

然而，请记住，处理非200 responses通常不是一个好主意，除非你真的知道自己在做什么。

有关更多信息，请参见:HTTP状态代码定义。

HttpErrorMiddleware settings

HTTPERROR_ALLOWED_CODES

默认: []

传递包含在此列表中的非200状态码的所有responses。

HTTPERROR_ALLOW_ALL

默认: False

传递所有responses，不管其状态代码如何。

OffsiteMiddleware

scrapy.spidermiddlewares.offsite.OffsiteMiddleware 这是一个类(class)

过滤spider覆盖的domains之外的url Requests。

这个middleware过滤掉所有主机名不在spider的allowed_domains属性中的request。列表中任何domain的所有子域都是允许的。例如，规则www.example.org也允许bob.www.example.org，但不允许www2.example.com或example.com。

当spider返回一个不属于spider覆盖的domain的request时，该middleware将log一条类似于此的debug消息:

DEBUG: Filtered offsite request to 'www.othersite.com': <GET http://www.othersite.com/some/page.html>

为了避免在log中填入太多垃圾信息，它只会为每个新过滤的domain打印其中一条消息。例如，如果过滤了对www.othersite.com的另一个request，则不会print log消息。但是，如果过滤了对someothersite.com的request，就会print一条消息(但只针对过滤的第一个request)。

如果spider没有定义allowed_domains属性，或者属性为空，那么offsite(离线)middleware将允许所有requests。

如果request设置了dont_filter属性，那么离线中间件将允许请求，即使其domain在允许domain中列出。

RefererMiddleware

scrapy.spidermiddlewares.referer.RefererMiddleware 这是一个类(class)

填充Request Referer header, 基于生成它的Response的URL。

RefererMiddleware settings

REFERER_ENABLED

版本0.15添加此功能

默认: True

是否启用referer middleware。

REFERRER_POLICY

版本1.4添加此功能

默认: 'scrapy.spidermiddlewares.referer.DefaultReferrerPolicy'

在填充Request “Referer” header时应用的referrer policy。

还可以使用特殊的'referrer策略' Request.meta键为每个request设置referrer policy。
具有与'REFERRER_POLICY' setting相同的可接受值。

REFERRER_POLICY的可接受值

  • 要么是scrapy.spidermiddlewares.referer.ReferrerPolicy子类的路径 - 自定义策略或内置策略之一(参见下面的类)，
  • 或一个标准的w3c定义的字符串值，
  • 或者是特殊的"scrapy-default"。
字符串值 类名(作为字符串)
"scrapy-default" (默认) scrapy.spidermiddlewares.referer.DefaultReferrerPolicy
“no-referrer” scrapy.spidermiddlewares.referer.NoReferrerPolicy
“no-referrer-when-downgrade” scrapy.spidermiddlewares.referer.NoReferrerWhenDowngradePolicy
“same-origin” scrapy.spidermiddlewares.referer.SameOriginPolicy
“origin” scrapy.spidermiddlewares.referer.OriginPolicy
“strict-origin” scrapy.spidermiddlewares.referer.StrictOriginPolicy
“origin-when-cross-origin” scrapy.spidermiddlewares.referer.OriginWhenCrossOriginPolicy
“strict-origin-when-cross-origin” scrapy.spidermiddlewares.referer.StrictOriginWhenCrossOriginPolicy
“unsafe-url” scrapy.spidermiddlewares.referer.UnsafeUrlPolicy

scrapy.spidermiddlewares.referer.DefaultReferrerPolicy 这是一个类(class)

“no-referrer-when-downgrade”的变体，如果parent request使用file://或s3:// scheme，则不发送“Referer”。

Scrapy的默认referrer策略(就像浏览器的w3c推荐值'[“no-referrer-when-downgrade”](https://www.w3.org/TR/referrer-policy/#referrer-policy-no-referrer-when-downgrade)'一样)
将从任何  http(s)://    发送非空的“Referer” header到任何   https:// URL，即使domain不同。

如果您想删除cross-domain requests的referrer信息，[“same-origin”](https://www.w3.org/TR/referrer-policy/#referrer-policy-same-origin)可能是更好的选择。

scrapy.spidermiddlewares.referer.NoReferrerPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-no-referrer

最简单的策略是“no-referrer”，它指定不将referrer信息连同从特定request客户端发出的requests一起发送到任何origin。header将被完全省略。

scrapy.spidermiddlewares.referer.NoReferrerWhenDowngradePolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-no-referrer-when-downgrade

“no-referrer-when-downgrade” policy将一个完整的URL连同从受tls保护的环境settings对象发出的requests发送到一个可能值得信任的URL，以及从不受tls保护的客户端发出的requests发送到任何origin。

另一方面，从受tls保护的客户端到不可靠的url的Requests将不包含referrer信息

如果没有指定policy，这是user agent的默认行为。

“no-referrer-when-downgrade” 策略是w3c推荐的默认设置，主要web浏览器都使用这种策略。

但是，它不是Scrapy的默认referrer策略(参见[DefaultReferrerPolicy](#))。

scrapy.spidermiddlewares.referer.SameOriginPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-same-origin

“same-origin”策略指定一个完整的URL, 在从特定request client发出same-origin requests时，作为referrer信息发送。

另一方面，Cross-origin requests将不包含referrer信息, 不会发送HTTP header。

scrapy.spidermiddlewares.referer.OriginPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-origin

“origin”策略指定在发出两个same-origin requests和来自特定request client的cross-origin requests时，只有请request client的ASCII序列化作为referrer信息发送

scrapy.spidermiddlewares.referer.StrictOriginPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-strict-origin

当发出requests时，“strict-origin”策略将request client的ASCII序列化:-从受tls保护的环境settings对象发送到可能值得信任的URL, 从不受tls保护的环境settings 对象发送到任何origin。

另一方面，从受tls保护的request clients到不值得信任的url的Requests将不包含referrer信息。不会发送Referer HTTP header。

scrapy.spidermiddlewares.referer.OriginWhenCrossOriginPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-origin-when-cross-origin

“origin-when-cross-origin”策略指定一个完整的URL， 当从特定的request client发出same-origin requests时将其剥离出来作为referrer使用，作为referrer信息发送。 当从特定的request client发出cross-origin requests时只有request client的ASCII序列化作为referrer信息发送。

scrapy.spidermiddlewares.referer.StrictOriginWhenCrossOriginPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-strict-origin-when-cross-origin

“strict-origin-when-cross-origin”策略指定一个完整的URL, 当从特定的request client发出same-origin requests时将其剥离出来作为referrer使用，作为referrer信息发送。 当发出cross-origin requests时只对request client进行ASCII序列化

  • 从tls保护的环境settings对象到可能值得信任的URL
  • 从非tls保护的环境settings对象到任何原origin。

另一方面，从受tls保护的clients到不可靠的url的Requests将不包含referrer信息。不会发送Referer HTTP header。

scrapy.spidermiddlewares.referer.UnsafeUrlPolicy 这是一个类(class)

https://www.w3.org/TR/referrer-policy/#referrer-policy-unsafe-url

“unsafe-url”策略指定一个完整的URL(可以作为referrer使用)与来自特定request client的cross-origin requests和same-origin requests一起发送。

注意:该策略的名称不会说谎;它是不安全的。该策略将泄漏origins和paths ，从tls保护资源到不安全的origins。仔细考虑为可能敏感的文档设置这样的策略的影响。

不建议使用“unsafe-url”策略。
