Extensions(扩展)

extensions框架提供了一种将您自己的自定义功能插入Scrapy的机制。

扩展只是在初始化扩展时在Scrapy启动时实例化的常规类。

Extension settings

extensions使用Scrapy settings来管理它们的settings，就像其他任何Scrapy代码一样。

extensions通常在其设置前加上自己的名称，以避免与现有(和未来)extensions冲突。例如，处理Google Sitemaps的假设extensions将使用GOOGLESITEMAP_ENABLED、GOOGLESITEMAP_DEPTH等settings。

加载和激活extensions

通过实例化extension类的单个实例，在启动时加载和激活Extensions。因此，所有extensions初始化代码都必须在类构造函数(__init__方法)中执行。

要使extension可用，请将其添加到Scrapy setting中的EXTENSIONS setting中。在EXTENSIONS中，每个extension都由一个字符串表示:extension的类名的完整Python路径。例如:

EXTENSIONS = {
    'scrapy.extensions.corestats.CoreStats': 500,
    'scrapy.extensions.telnet.TelnetConsole': 500,
}

可以看到，EXTENSIONS setting是一个dict，其中键是extension路径，它们的值是数字, order定义extension加载顺序。EXTENSIONS setting与Scrapy中定义的EXTENSIONS_BASE setting合并(不打算被覆盖)，然后按顺序进行排序，以获得启用extensions的最终排序列表。

由于extensions通常不相互依赖，因此它们的加载顺序在大多数情况下都无关紧要。这就是为什么EXTENSIONS_BASE setting用相同的顺序(0)定义所有extensions的原因。但是，如果您需要添加依赖于已加载的其他扩展的扩展，那么可以利用这个特性。

可用、启用和禁用extensions

并不是所有可用的extensions都将被启用。其中一些通常取决于特定的setting。例如，HTTP Cache extension在默认情况下是可用的，但是要想禁用，除非设置了HTTPCACHE_ENABLED setting。

禁用一个extension

为了禁用默认启用的extension(即。那些包含在EXTENSIONS_BASE setting中的)您必须将其顺序设置为None。例如:

EXTENSIONS = {
    'scrapy.extensions.corestats.CoreStats': None,
}

编写自己的extension

每个extension都是Python类。Scrapy扩展的主要入口点(这也包括middlewares和pipelines)是from_crawler类方法，它接收一个Crawler实例。通过Crawler对象，您可以访问settings, signals, stats，还可以控制crawler行为。

通常，extensions连接到signals并执行由signals触发的任务。

最后，如果from_crawler方法引发NotConfigured异常，那么extension将被禁用。否则，将启用extension。

示例extension

在这里，我们将实现一个简单的extension来说明上一节中描述的概念。此extension将每次log一条消息:

  • spider打开的时候
  • spider关闭的时候
  • 一个特定数量的items被抓取

extension将通过MYEXT_ENABLED setting启用，items数量将通过MYEXT_ITEMCOUNT setting指定。

以下是该extension的代码:

import logging
from scrapy import signals
from scrapy.exceptions import NotConfigured

logger = logging.getLogger(__name__)

class SpiderOpenCloseLogging(object):

    def __init__(self, item_count):
        self.item_count = item_count
        self.items_scraped = 0

    @classmethod
    def from_crawler(cls, crawler):
        # 首先检查extension是否应该被启用，否则抛出notconfigure
        if not crawler.settings.getbool('MYEXT_ENABLED'):
            raise NotConfigured

        # 从settings中获取items的数量
        item_count = crawler.settings.getint('MYEXT_ITEMCOUNT', 1000)

        # 实例化extension对象
        ext = cls(item_count)

        # 将extension对象连接到signals
        crawler.signals.connect(ext.spider_opened, signal=signals.spider_opened)
        crawler.signals.connect(ext.spider_closed, signal=signals.spider_closed)
        crawler.signals.connect(ext.item_scraped, signal=signals.item_scraped)

        # 返回extension对象
        return ext

    def spider_opened(self, spider):
        logger.info(u"打开spider %s", spider.name)

    def spider_closed(self, spider):
        logger.info(u"关闭spider %s", spider.name)

    def item_scraped(self, item, spider):
        self.items_scraped += 1
        if self.items_scraped % self.item_count == 0:
            logger.info(u"已经抓取了 %d items", self.items_scraped)

内置extensions参考

通用extensions

Log Stats扩展

scrapy.extensions.logstats.LogStats 这是一个类(class)

log基本的stats，如爬过的页面和已经抓取的items。

Core Stats扩展

scrapy.extensions.corestats.CoreStats 这是一个类(class)

启用核心统计信息的集合，前提是启用了stats collection(参见Stats Collection)。

Telnet console扩展

scrapy.extensions.telnet.TelnetConsole 这是一个类(class)

提供一个telnet console，用于在当前正在运行的Scrapy进程内部进入Python解释器，这对于调试非常有用。

telnet console必须通过TELNETCONSOLE_ENABLED setting启用，服务器将侦听TELNETCONSOLE_PORT中指定的端口。

内存使用扩展

scrapy.extensions.memusage.MemoryUsage 这是一个类(class)

这个扩展在Windows中不起作用。

监视运行spider的Scrapy进程使用的内存:

  1. 当它超过某个值时发送通知电子邮件
  2. 当它超过某个值时关闭spider

当达到某个警告值(MEMUSAGE_WARNING_MB)和达到最大值(MEMUSAGE_LIMIT_MB)时，可以触发电子邮件通知，这也会导致spider关闭和Scrapy进程终止。

这个extension是通过MEMUSAGE_ENABLED setting启用的，可以配置如下settings:

  • MEMUSAGE_LIMIT_MB
  • MEMUSAGE_WARNING_MB
  • MEMUSAGE_NOTIFY_MAIL
  • MEMUSAGE_CHECK_INTERVAL_SECONDS

内存调试器扩展

scrapy.extensions.memdebug.MemoryDebugger 这是一个类(class)

调试内存使用的extension。它收集关于:

  • 未被Python垃圾收集器收集的对象
  • 那些不应该存活的对象。有关更多信息，请参阅使用trackref调试内存泄漏

要启用此extension，请打开MEMDEBUG_ENABLED setting。信息将存储在stats中。

关闭spider扩展

scrapy.extensions.closespider.CloseSpider 这是一个类(class)

当满足某些条件时，使用每个条件的特定关闭原因自动关闭spider。

关闭spider的条件可以通过以下settings配置:

  • CLOSESPIDER_TIMEOUT
  • CLOSESPIDER_ITEMCOUNT
  • CLOSESPIDER_PAGECOUNT
  • CLOSESPIDER_ERRORCOUNT

CLOSESPIDER_TIMEOUT

默认: 0

指定秒数的整数。如果spider的打开时间超过该秒数，它将以closespider_itemcount的原因自动关闭。如果为零(或未设置)，则不会在超时时关闭spider。

CLOSESPIDER_ITEMCOUNT

默认: 0

指定若干items的整数。如果spider抓取的数量超过这个数量，并且这些items通过pipeline传递，那么spider将以closespider_itemcount的原因关闭。当前在downloader队列(直到CONCURRENT_REQUESTS requests)中的请求仍然在处理中。如果为零(或未设置)，spider将不会按传递items的数量关闭。

CLOSESPIDER_PAGECOUNT

版本0.11添加此功能

默认: 0

指定要抓取的responses的最大数量的整数。如果抓取的次数多于此数，则关闭spider的原因为closespider_pagecount。如果为零(或未设置)，spider不会因为抓取的次数而关闭。

CLOSESPIDER_ERRORCOUNT

版本0.11添加此功能

默认: 0

在关闭spider之前指定要接收的最大错误数的整数。如果spider生成的错误数量超过这个数量，它将以closespider_errorcount的原因关闭。如果为零(或未设置)，spider将不会被错误数关闭。

StatsMailer扩展

scrapy.extensions.statsmailer.StatsMailer 这是一个类(class)

这个简单的扩展可以用于在domain完成抓取(包括收集到的Scrapy stats)之后发送电子邮件通知。电子邮件将发送到STATSMAILER_RCPTS setting中指定的所有收件人。

调试扩展

堆栈跟踪转储扩展

scrapy.extensions.debug.StackTraceDump 这是一个类(class)

当接收到SIGQUIT或SIGUSR2信号时，转储关于正在运行的进程的信息。转储的信息如下:

  1. engine status(使用scrapy.utils.engine.get_engine_status())
  2. live references(看使用trackref调试内存泄漏)
  3. 所有线程的堆栈跟踪

在转储堆栈跟踪和engine status之后，Scrapy进程继续正常运行。

这个扩展只能在兼容posix的平台上运行。，因为SIGQUIT和SIGUSR2信号在Windows上不可用。

发送Scrapy的SIGQUIT信号的方式至少有两种:

  1. 按ctrl — 当一个Scrapy程序正在运行时(仅在Linux下运行?)

  2. 通过运行这个命令(假设<pid>是Scrapy进程的进程id):

    kill -QUIT <pid>

调试器扩展

scrapy.extensions.debug.Debugger 这是一个类(class)

当接收到SIGUSR2信号时，在正在运行的Scrapy进程中调用Python调试器。调试器退出后，Scrapy进程继续正常运行。

有关更多信息，请参阅Python调试器。

这个扩展只能在兼容posix的平台上运行。(而不是Windows)。
