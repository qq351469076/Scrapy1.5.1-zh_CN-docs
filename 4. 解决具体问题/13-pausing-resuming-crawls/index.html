作业: 暂停和恢复爬虫

有时候，对于大型站点来说，暂停爬行并能够在稍后恢复它们是可取的。

Scrapy通过提供以下功能来支持这种开箱即用的功能:

  • 在磁盘上持久存储计划请求的scheduler
  • 一个重复的过滤器，它将访问的请求保存在磁盘上
  • 在分批之间保持一些spider状态(键/值对)持久的扩展

作业目录

要启用持久性支持，只需通过JOBDIR setting定义一个作业目录。这个目录将用于存储所有需要的数据，以保持单个作业的状态(即。一个spider运行)。需要注意的是，这个目录不能被不同的spider共享，甚至不能被同一个spider的不同作业/运行共享，因为它是用来存储单个作业的状态的。

使用方法

要启动支持持久化的支持的spider，运行它如下:

scrapy crawl somespider -s JOBDIR=crawls/somespider-1

然后，您可以在任何时候安全地停止spdier(按Ctrl-C或发送一个信号)，然后通过发出相同的命令恢复它:

scrapy crawl somespider -s JOBDIR=crawls/somespider-1

保持批次之间的持久状态

有时您需要在暂停/恢复批次之间保持一些持久的spider state。你可以用spider.state属性, 它应该是dict。有一个内置的扩展，当spider启动和停止时，它负责从作业目录序列化、存储和加载该属性。

下面是一个使用spider state的callback示例(为了简洁起见，省略了其他spider代码):

def parse_item(self, response):
    # 在这里解析item
    self.state['items_count'] = self.state.get('items_count', 0) + 1

持久性问题

如果您想要使用Scrapy持久性支持，需要记住以下几点:

Cookies过期

Cookies可能会过期。因此，如果您不迅速恢复spider，那么计划中的请求可能不再有效。如果spider不依赖cookie，这就不是问题。

Request序列化

Requests必须由pickle模块序列化，以便持久化工作，因此应该确保requests是可序列化的。

这里最常见的问题是对不能持久的request callback使用lambda函数。

例如，这是行不通的:

def some_callback(self, response):
    somearg = 'test'
    return scrapy.Request('http://www.example.com', callback=lambda r: self.other_callback(r, somearg))

def other_callback(self, response, somearg):
    print "the argument passed is:", somearg

但这可以:

def some_callback(self, response):
    somearg = 'test'
    return scrapy.Request('http://www.example.com', callback=self.other_callback, meta={'somearg': somearg})

def other_callback(self, response):
    somearg = response.meta['somearg']
    print "the argument passed is:", somearg

如果希望log无法序列化的requests，可以在项目的settings页面中将SCHEDULER_DEBUG设置设置为True。默认为False。
