调试内存泄漏

在Scrapy中，Requests、Responses和Items等对象的生命周期是有限的:它们被创建、使用一段时间，最后被销毁。

在所有这些对象中，Request可能是生命周期最长的一个，因为它在Scheduler队列中等待，直到处理它的时候。有关更多信息，请参阅体系结构概述。

由于这些Scrapy对象有一个(相当长的)生命周期，因此在没有适当释放它们的情况下，在内存中积累它们总是有风险的，从而导致所谓的“内存泄漏”。

为了帮助调试内存泄漏，Scrapy提供了一个用于跟踪对象引用的内置机制，称为trackref，您还可以使用名为Guppy的第三方库进行更高级的内存调试(有关更多信息，请参阅下面)。这两种机制都必须从Telnet Console使用。

内存泄漏的常见原因

Scrapy开发人员经常(有时是偶然的，有时是故意的)会传递Requests(例如，使用meta属性或request callback函数)中引用的对象，这有效地将这些引用对象的生命周期限制到Requests的生命周期。到目前为止，这是Scrapy项目中导致内存泄漏的最常见原因，而且对于新手来说很难调试。

在大型项目中，spider通常由不同的人编写，其中一些爬行器可能会“泄漏”，因此，当其他(写得很好的)spider并发运行时，其他的spider也会受到影响，这反过来又会影响整个爬行过程。

如果您没有正确释放(先前分配的)资源，泄漏也可能来自您编写的自定义middleware、pipeline或extension。例如，如果在每个进程上运行多个spider，那么在spider_opened上分配资源而不在spider_closed上释放资源可能会导致问题。

太多Requests?

默认情况下，Scrapy将request队列保存在内存中;它包括Request对象和Request属性中引用的所有对象(例如meta)。虽然不一定是泄漏，但这会占用大量内存。启用持久作业队列可以帮助控制内存使用。

使用trackref调试内存泄漏

trackref是Scrapy提供的一个模块，用于调试最常见的内存泄漏情况。它基本上跟踪对所有活动的Requests、Responses、Item和Selector对象的引用。

您可以进入telnet console，并使用prefs()函数检查当前有多少对象(上面提到的类)在运行，prefs()函数是print_live_refs()函数的别名:

telnet localhost 6023

>>> prefs()
Live References

ExampleSpider                       1   oldest: 15s ago
HtmlResponse                       10   oldest: 1s ago
Selector                            2   oldest: 0s ago
FormRequest                       878   oldest: 7s ago

如您所见，该报告还显示了每个类中oldest(最古老)对象的“age”。如果您在每个进程中运行多个spider，那么您可以通过查看最老的request或response来确定哪个spider正在泄漏。您可以使用get_oldest()函数(从telnet console)获得每个类的最古老对象。

跟踪哪些对象?

trackref跟踪的对象都来自这些类(及其所有子类):

  • scrapy.http.Request
  • scrapy.http.Response
  • scrapy.item.Item
  • scrapy.selector.Selector
  • scrapy.spiders.Spider

一个真实的例子

让我们来看一个内存泄漏的具体例子。假设我们有一些spider，它们的路线和这个类似:

return Request("http://www.somenastyspider.com/product.php?pid=%d" % product_id,
    callback=self.parse, meta={referer: response})

这一行是在request中传递一个response引用，该引用有效地将响response生命周期与request的生命周期联系起来，这肯定会导致内存泄漏。

让我们看看如何通过使用trackref工具来发现原因(当然不是事先知道的)。

爬虫程序运行几分钟后，我们注意到它的内存使用量增加了很多，我们可以进入它的telnet console并检查活动引用:

>>> prefs()
Live References

SomenastySpider                     1   oldest: 15s ago
HtmlResponse                     3890   oldest: 265s ago
Selector                            2   oldest: 0s ago
Request                          3878   oldest: 250s ago

有这么多的活动responses(而且它们太老了)这一事实无疑是值得怀疑的，因为与Requests相比，responses的生命周期应该相对较短。responses的数量与requests的数量类似，所以它们在某种程度上是联系在一起的。现在，我们可以检查spider的代码，以发现生成泄漏的恶意代码行(在requests中传递response引用)。

有时候关于活动对象的额外信息可能会有帮助。让我们看看最古老的response:

>>> from scrapy.utils.trackref import get_oldest
>>> r = get_oldest('HtmlResponse')
>>> r.url
'http://www.somenastyspider.com/product.php?pid=123'

如果你想遍历所有对象，你可以使用scrapy.utils.trackref.iter_all()函数来代替获取get_oldest:

>>> from scrapy.utils.trackref import iter_all
>>> [r.url for r in iter_all('HtmlResponse')]
['http://www.somenastyspider.com/product.php?pid=123',
 'http://www.somenastyspider.com/product.php?pid=584',
...

太多的spiders?

如果您的项目有太多并行执行的spider，那么prefs()的输出可能难以读取。因此，该函数有一个ignore参数，可以用来忽略特定的类(以及它的所有子类)。例如，这不会显示任何对spider的活动引用:

>>> from scrapy.spiders import Spider
>>> prefs(ignore=Spider)

scrapy.utils.trackref模块

下面是trackref模块中提供的可用函数。

scrapy.utils.trackref.object_ref 这是一个类(class)

如果您想使用trackref模块跟踪活动实例，可以从这个类(而不是对象)继承。

scrapy.utils.trackref.print_live_refs(class_name, ignore=NoneType)

按类名分组，打印实时引用的报告。

参数:

  • ignore (类或类元组) – 如果给定，来自指定类(或类的元组)的所有对象都将被忽略。

scrapy.utils.trackref.get_oldest(class_name)

返回具有给定类名的最古老对象，如果没有找到，则返回None。首先使用print_live_refs()获得每个类名跟踪的所有活动对象的列表。

scrapy.utils.trackref.iter_all(class_name)

在具有给定类名的所有对象上返回迭代器，如果没有找到类名，则返回None。首先使用print_live_refs()获得每个类名跟踪的所有活动对象的列表。

用Guppy调试内存泄漏

trackref提供了一种非常方便的机制来跟踪内存泄漏，但是它只跟踪更可能导致内存泄漏的对象(Requests、Responses、Items和Selectors)。然而，在其他情况下，内存泄漏可能来自其他(或多或少有些模糊的)对象。如果这是您的情况，并且您无法使用trackref找到泄漏，那么您还有另一个资源:Guppy库。如果您正在使用Python3，请参阅用muppy调试内存泄漏。

如果使用pip，可以使用以下命令安装Guppy:

pip install guppy

没有泄露的泄露

有时，您可能会注意到Scrapy进程的内存使用只会增加，但不会减少。不幸的是，即使Scrapy和项目都没有泄漏内存，也可能发生这种情况。这是由于众所周知的Python问题，在某些情况下，它可能不会将释放的内存返回到操作系统。有关这个问题的更多信息，请参阅:

  • Python的内存管理
  • Python的内存管理第2部分
  • Python的内存管理第3部分

本文中详细介绍了Evan Jones提出的改进，在Python 2.5中得到了合并，但这只是减少了问题，并没有完全解决问题。引用本文:

不幸的是，这个补丁只能在不再分配对象的情况下释放一个竞技场。这意味着碎片化是一个大问题。一个应用程序可以有许多兆字节的空闲内存，分散在所有的竞技场中，但是它不能释放任何一个。这是所有内存分配器都会遇到的问题。解决这个问题的唯一方法是转移到压缩垃圾收集器，它能够在内存中移动对象。这需要对Python解释器进行重大更改。

为了保持合理的内存消耗，您可以将job分割为几个较小的job，或者启用持久作业队列并不时地停止/启动spider。
